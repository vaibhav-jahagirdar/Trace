# Final Evaluation Report Structure

This document defines the exact structure Stage 1 must produce at the end of evaluation, and includes one complete worked example.

This is the contract between the LLM's output and `ResumeEvaluationReport` in PostgreSQL. Every field below maps to a stored column or a joined table row. Field order within each section reflects required generation order — reasoning and evidence fields are always defined before the conclusion fields that depend on them, because a model asked to fill a conclusion field before its supporting evidence field commits to the answer before finishing the reasoning.

---

## Governing Rules for This Document

- All terminology (`CONFIRMED` / `UNCONFIRMED` / `MISSING` / `UNDETERMINABLE`) follows `definitions.md` exactly. Do not use synonyms.
- Every `supporting_claim_ids` field references stable `claim_id` values from the Candidate Extraction output — never free-text quotes, never invented citations. If no extracted claim supports a conclusion, the conclusion must not be made (`constraints.md`, Grounding).
- Every scored/classified field is preceded by the evidence fields that justify it, never followed by them.
- **`resume_match_score` is not generated by the LLM.** It is computed deterministically downstream as the sum of the seven bucket scores, after generation. This avoids relying on single-pass LLM arithmetic across seven numbers to self-consistently sum to a total. The field exists in the stored report for query convenience, but the model must never attempt to output it.
- Confidence values are always one of `HIGH` / `MEDIUM` / `LOW`, per `definitions.md`. Confidence reflects certainty of the assessment, not truthfulness of the underlying claim.

## 1. Metadata

```yaml
metadata:
  schema_version: string
  job_id: uuid            # join key into jobs table
  application_id: uuid    # join key into job_applications table
  extraction_id: uuid     # join key into the CandidateExtraction row this report was built from
  model: string
  timestamp: datetime
  evaluation_duration_ms: integer
```

`job_id` / `application_id` / `extraction_id` were absent from the earlier draft and are required — without them this report cannot be joined back to what it evaluated.

---

## 2. Role Fit

Evaluates `evaluation_policy.md` priorities #2 and #3 (Role Alignment, Responsibility Alignment) as first-class scored fields, not buried prose inside Bucket 1. This was a gap in the earlier draft — `Role Fit` was referenced in `contract.md`'s `output_requirements` and `secondary_tiebreakers` but never actually defined anywhere.

```yaml
role_fit:
  role_alignment_evidence: list[claim_id]
  role_alignment_summary: string          # ≤ 25 words
  role_alignment: HIGH | MEDIUM | LOW | UNDETERMINABLE

  responsibility_alignment_evidence: list[claim_id]
  responsibility_alignment_summary: string  # ≤ 25 words
  responsibility_alignment: HIGH | MEDIUM | LOW | UNDETERMINABLE

  domain_alignment_evidence: list[claim_id]
  domain_alignment_summary: string          # ≤ 25 words
  domain_alignment: HIGH | MEDIUM | LOW | UNDETERMINABLE
```

---

## 3. Bucket Scores

Exactly the seven buckets from `scoring_rubrics.md` / `contract.md`. Field order within each bucket: evidence → summary → confidence → score. `max_score` is a constant per bucket, not model-generated.

```yaml
bucket_scores:
  primary_evidence:
    source: WORK | PROJECT           # which one was primary, per exception_policy.md
    supporting_claim_ids: list[claim_id]
    summary: string                  # ≤ 40 words
    confidence: HIGH | MEDIUM | LOW
    score: float                     # 0–35

  secondary_evidence:
    source: WORK | PROJECT
    supporting_claim_ids: list[claim_id]
    summary: string
    confidence: HIGH | MEDIUM | LOW
    score: float                     # 0–20

  concept_alignment:
    supporting_claim_ids: list[claim_id]
    summary: string
    confidence: HIGH | MEDIUM | LOW
    score: float                     # 0–15

  technology_alignment:
    mandatory_technologies_present: boolean
    supporting_claim_ids: list[claim_id]
    summary: string
    confidence: HIGH | MEDIUM | LOW
    score: float                     # 0–8

  technical_claim_quality:
    supporting_claim_ids: list[claim_id]
    summary: string
    confidence: HIGH | MEDIUM | LOW
    score: float                     # 0–8

  recruiter_alignment:
    supporting_claim_ids: list[claim_id]
    summary: string
    confidence: HIGH | MEDIUM | LOW
    score: float                     # 0–10

  supporting_signals:
    supporting_claim_ids: list[claim_id]
    summary: string
    confidence: HIGH | MEDIUM | LOW
    score: float                     # 0–4
```

`resume_match_score` (0–100) = sum of the seven `score` values above, computed by the backend after generation — see Governing Rules.

## 4. Requirement Analysis

**Merged and restructured from the earlier draft.** The prior version had `RequirementAnalysis`, `TechnologyAnalysis`, and `ConceptAnalysis` as three separate top-level sections representing overlapping data in inconsistent shapes. This single section replaces all three, nested by priority tier and then by type — mirroring the job-context DTO shape (`requirements.mandatory.technologies` / `.concepts`, etc.) so the report structure and the job-context structure stay symmetric.

```yaml
requirement_analysis:
  mandatory:
    technologies: list[RequirementAssessment]
    concepts: list[RequirementAssessment]
  preferred:
    technologies: list[RequirementAssessment]
    concepts: list[RequirementAssessment]
  bonus:
    technologies: list[RequirementAssessment]
    concepts: list[RequirementAssessment]
```

Where `RequirementAssessment` is:

```yaml
RequirementAssessment:
  name: string
  status: CONFIRMED | UNCONFIRMED | MISSING
  supporting_claim_ids: list[claim_id]   # empty when status is MISSING
  note: string                            # ≤ 20 words, empty allowed
```

Every mandatory/preferred/bonus × technology/concept item from the job context appears exactly once, in exactly one of these six lists, classified per `definitions.md`.

## 5. Experience Analysis

```yaml
experience_analysis:
  primary_source: WORK | PROJECT
  relevant_experiences:
    - experience_id: claim_id            # references a work_experience or project entity
      relevance_evidence: list[claim_id]
      relevance_summary: string          # ≤ 30 words
  experience_summary: string             # ≤ 50 words, overall synthesis only — no new claims introduced here
```

---

## 6. Project Analysis

Feeds directly into `verification_plan` and GitHub Stage 2.

```yaml
project_analysis:
  prioritized_projects:
    - project_id: claim_id
      relevance_evidence: list[claim_id]
      summary: string                    # ≤ 30 words
      verification_value: HIGH | MEDIUM | LOW
      priority: integer                  # 1 = highest
  ignored_projects: list[project_id]     # projects extracted but not job-relevant; no penalty implied
```

---

## 7. Recruiter Rubric Alignment

Reflects the recruiter's actual configured dimensions — dynamic, not a hardcoded taxonomy. This replaces the fixed `BucketScores`-style categories that conflicted with the configurable `evaluation_dimensions` / `evidence_categories` / `success_signals` tables.

```yaml
recruiter_rubric:
  evaluation_dimensions:
    - code: string                       # from evaluation_dimensions.code
      supporting_claim_ids: list[claim_id]
      summary: string                    # ≤ 25 words
      alignment: HIGH | MEDIUM | LOW | UNDETERMINABLE
  evidence_categories:
    - code: string
      supporting_claim_ids: list[claim_id]
      summary: string
      alignment: HIGH | MEDIUM | LOW | UNDETERMINABLE
  success_signals:
    - code: string
      supporting_claim_ids: list[claim_id]
      summary: string
      alignment: HIGH | MEDIUM | LOW | UNDETERMINABLE
```

One entry per item the recruiter actually configured for this job — no invented dimensions, per `evaluation_policy.md` §8–10.

---

## 8. Strengths & Gaps

```yaml
strengths:
  - category: string
    supporting_claim_ids: list[claim_id]
    summary: string                      # ≤ 25 words

gaps:
  - category: string
    supporting_claim_ids: list[claim_id]  # empty allowed for MISSING-type gaps
    summary: string                       # ≤ 25 words
    impact: HIGH | MEDIUM | LOW
```

---

## 9. Verification Plan

Consumed directly by GitHub Stage 2.

```yaml
verification_plan:
  verification_targets:
    - claim_id: claim_id
      claim_summary: string              # ≤ 20 words, restated for readability
      related_project_id: claim_id | null
      importance: CRITICAL | HIGH | MEDIUM
      why_verify: string                 # ≤ 25 words — what specifically to check in code
  repository_strategy: string            # ≤ 40 words — overall approach for Stage 2, not per-claim detail
```

---

## 10. Technical Outlier

Per `scoring_rubrics.md` — never affects score, may override repository-analysis gating.

```yaml
technical_outlier:
  is_outlier: boolean
  supporting_claim_ids: list[claim_id]   # required when is_outlier is true
  justification: string                  # ≤ 40 words, empty when is_outlier is false
  missing_requirements: list[string]     # mandatory items this candidate lacks despite outlier status
  repository_analysis_recommended: boolean
```

---

## 11. Confidence

```yaml
confidence:
  extraction_quality: HIGH | MEDIUM | LOW   # confidence in the upstream extraction this report was built from
  scoring_quality: HIGH | MEDIUM | LOW      # confidence in this evaluation's own conclusions
  overall: HIGH | MEDIUM | LOW              # min(extraction_quality, scoring_quality) — computed downstream, not by the LLM
```

---

## 12. Overall Evaluation

```yaml
overall:
  overall_role_fit: EXCEPTIONAL | STRONG | GOOD | MODERATE | WEAK | POOR
  repository_priority: CRITICAL | HIGH | MEDIUM | LOW
```

`resume_match_score` and `proceed_to_repository_analysis` are deliberately **not** here — the former is computed downstream (see Governing Rules), and the latter is not a separate field at all: downstream treats `repository_priority != LOW` as the default proceed signal, with `technical_outlier.repository_analysis_recommended` as the explicit override for outliers. Keeping one gating signal per situation avoids two boolean-shaped fields that can silently disagree with each other.

---

## 13. Executive Summary

Human-facing only. Must not introduce any claim not already grounded elsewhere in the report.

```yaml
executive_summary:
  recruiter_summary: string        # ≤ 60 words
  top_strengths: list[string]      # ≤ 3 items, each ≤ 12 words
  primary_risks: list[string]      # ≤ 3 items, each ≤ 12 words
```

`recommendation` is removed from the earlier draft — Stage 1 does not make hiring recommendations (`role.md`, `constraints.md`); a free-text "recommendation" field invites exactly that. `repository_priority` in section 12 already carries the actionable next-step signal.

# Worked Example

Candidate: the same extraction example from the Candidate Extraction Contract (Backend Engineer, ABC Technologies, OrgOps project).

Job (assumed for this example): Backend Engineer role requiring Node.js (mandatory), PostgreSQL (mandatory), RBAC (mandatory concept), Redis (preferred), Refresh Token Rotation (preferred concept), Docker (bonus).

```yaml
metadata:
  schema_version: "1.0"
  job_id: "3fa8-...-b21e"
  application_id: "9c14-...-77af"
  extraction_id: "b502-...-114d"
  model: "claude-sonnet-5"
  timestamp: "2026-07-11T10:22:00Z"
  evaluation_duration_ms: 4210

role_fit:
  role_alignment_evidence: ["claim_004"]
  role_alignment_summary: "Candidate's current title and responsibilities directly match Backend Engineer scope."
  role_alignment: HIGH

  responsibility_alignment_evidence: ["claim_004", "claim_009"]
  responsibility_alignment_summary: "Built authentication and org-management services — core backend ownership responsibilities."
  responsibility_alignment: HIGH

  domain_alignment_evidence: []
  domain_alignment_summary: "No domain/industry claims present to compare against job context."
  domain_alignment: UNDETERMINABLE

bucket_scores:
  primary_evidence:
    source: WORK
    supporting_claim_ids: ["claim_004", "claim_005", "claim_006"]
    summary: "2 years professional backend work with specific auth implementation claims (JWT, refresh rotation, replay detection)."
    confidence: HIGH
    score: 26

  secondary_evidence:
    source: PROJECT
    supporting_claim_ids: ["claim_009", "claim_010"]
    summary: "OrgOps project reinforces backend claims with RBAC and audit logging, directly job-relevant."
    confidence: HIGH
    score: 16

  concept_alignment:
    supporting_claim_ids: ["claim_006", "claim_011"]
    summary: "RBAC (mandatory) and Refresh Token Rotation (preferred) both explicitly implemented, not just mentioned."
    confidence: HIGH
    score: 13

  technology_alignment:
    mandatory_technologies_present: true
    supporting_claim_ids: ["claim_004", "claim_009"]
    summary: "Node.js and PostgreSQL both confirmed with implementation context, not bare mentions."
    confidence: HIGH
    score: 7

  technical_claim_quality:
    supporting_claim_ids: ["claim_005", "claim_006"]
    summary: "Specific mechanisms named (replay detection, token rotation) rather than generic 'built auth system' phrasing."
    confidence: HIGH
    score: 7

  recruiter_alignment:
    supporting_claim_ids: ["claim_006", "claim_011"]
    summary: "Strong match on recruiter-defined concept-depth evaluation dimension; limited signal on collaboration-related dimension."
    confidence: MEDIUM
    score: 7

  supporting_signals:
    supporting_claim_ids: ["claim_012"]
    summary: "B.Tech Computer Science — relevant baseline qualification, no differentiation beyond that."
    confidence: HIGH
    score: 2

requirement_analysis:
  mandatory:
    technologies:
      - name: "Node.js"
        status: CONFIRMED
        supporting_claim_ids: ["claim_004"]
        note: "Used across both work experience and project."
      - name: "PostgreSQL"
        status: CONFIRMED
        supporting_claim_ids: ["claim_009"]
        note: "Explicit in OrgOps project."
    concepts:
      - name: "RBAC"
        status: CONFIRMED
        supporting_claim_ids: ["claim_010"]
        note: "Explicitly implemented in OrgOps."
  preferred:
    technologies:
      - name: "Redis"
        status: CONFIRMED
        supporting_claim_ids: ["claim_005", "claim_009"]
        note: "Used in both work experience and project."
    concepts:
      - name: "Refresh Token Rotation"
        status: CONFIRMED
        supporting_claim_ids: ["claim_006"]
        note: "Explicit implementation claim."
  bonus:
    technologies:
      - name: "Docker"
        status: MISSING
        supporting_claim_ids: []
        note: "Not present in resume or application."
    concepts: []

experience_analysis:
  primary_source: WORK
  relevant_experiences:
    - experience_id: "claim_004"
      relevance_evidence: ["claim_004", "claim_005", "claim_006"]
      relevance_summary: "Directly relevant backend role with matching technology and concept claims."
  experience_summary: "Two years of directly relevant backend engineering work at ABC Technologies, closely matching this job's stack and responsibilities."

project_analysis:
  prioritized_projects:
    - project_id: "claim_009"
      relevance_evidence: ["claim_009", "claim_010"]
      summary: "OrgOps: RBAC, audit logging, and DB transactions — high-value verification target."
      verification_value: HIGH
      priority: 1
  ignored_projects: []

recruiter_rubric:
  evaluation_dimensions:
    - code: "CONCEPT_DEPTH"
      supporting_claim_ids: ["claim_006", "claim_010"]
      summary: "Concept implementations are specific and mechanism-level, not surface mentions."
      alignment: HIGH
    - code: "COLLABORATION_EVIDENCE"
      supporting_claim_ids: []
      summary: "No claims describing cross-functional or team collaboration present."
      alignment: UNDETERMINABLE
  evidence_categories:
    - code: "CODE_VERIFIABLE_CLAIMS"
      supporting_claim_ids: ["claim_006", "claim_010"]
      summary: "Multiple claims are specific enough to check directly against source code."
      alignment: HIGH
  success_signals:
    - code: "OWNERSHIP_INDICATORS"
      supporting_claim_ids: ["claim_009"]
      summary: "Sole ownership language used for OrgOps project."
      alignment: MEDIUM

strengths:
  - category: "Technical Claim Specificity"
    supporting_claim_ids: ["claim_006"]
    summary: "Replay detection and token rotation named as explicit mechanisms, not generic 'secure auth.'"
  - category: "Mandatory Requirement Coverage"
    supporting_claim_ids: ["claim_004", "claim_009", "claim_010"]
    summary: "All three mandatory requirements confirmed with implementation-level evidence."

gaps:
  - category: "Collaboration Evidence"
    supporting_claim_ids: []
    summary: "No claims address teamwork, cross-functional work, or communication."
    impact: LOW

verification_plan:
  verification_targets:
    - claim_id: "claim_006"
      claim_summary: "Refresh token rotation with replay detection."
      related_project_id: "claim_004"
      importance: CRITICAL
      why_verify: "Check for token blacklist/rotation logic and replay-guard middleware in the auth service."
    - claim_id: "claim_010"
      claim_summary: "RBAC and audit logging in OrgOps."
      related_project_id: "claim_009"
      importance: HIGH
      why_verify: "Check for role-based middleware and an audit log table/write path."
  repository_strategy: "Prioritize the OrgOps repository; focus on auth middleware and RBAC enforcement paths first."

technical_outlier:
  is_outlier: false
  supporting_claim_ids: []
  justification: ""
  missing_requirements: []
  repository_analysis_recommended: true

confidence:
  extraction_quality: HIGH
  scoring_quality: HIGH
  overall: HIGH

overall:
  overall_role_fit: STRONG
  repository_priority: HIGH

executive_summary:
  recruiter_summary: "Strong backend candidate with confirmed mandatory tech/concept coverage and specific, verifiable implementation claims. No collaboration evidence present."
  top_strengths:
    - "All mandatory requirements confirmed with implementation depth"
    - "Specific, mechanism-level security implementation claims"
  primary_risks:
    - "No evidence of collaborative or cross-functional work"
```

`resume_match_score` for this example, computed downstream (not by the LLM): 26 + 16 + 13 + 7 + 7 + 7 + 2 = **78** — "Good alignment, competitive repository analysis candidate" per `scoring_rubrics.md`'s calibration band.
